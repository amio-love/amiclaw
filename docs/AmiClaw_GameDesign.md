# AmiClaw BombSquad：人机协作拆弹游戏设计

> **平台定位**：BombSquad (`bombsquad.amio.fans`) 是 AmiClaw 平台 (`claw.amio.fans`) 上的第一款人机协作游戏。AmiClaw 是一个承载多种 agent+human 玩法的平台，BombSquad 是其第一个验证产品。

> **一句话概念**：Keep Talking and Nobody Explodes 的 AI 版——你是拆弹手，你的 AI 是读手册的专家。用语音实时沟通，看谁拆得最快。

## 1. 核心设计

### 1.1 角色分工

**玩家（拆弹手）**：在游戏网页中看到一个 2D 谜题场景。需要用语音向 agent 描述看到的内容，并根据 agent 的指导执行操作。考验表达能力、操作速度、和对 agent 建议的判断力。

**Agent（手册专家）**：阅读一份复杂的操作手册（一个网页链接）。根据玩家的语音描述，在手册中检索对应规则，通过语音告诉玩家该怎么操作。考验 AI 的理解能力、检索能力、主动引导能力。

### 1.2 核心体验循环

```markdown
玩家看到场景 → 语音描述给 agent → agent 查手册 → 语音告诉玩家怎么做 → 玩家执行操作 → 场景变化 → 新一轮循环
```

每一轮循环中，好玩的地方在于：

- **描述的精确度**：你说“红色按钮”，但场景里有三个红色按钮
- **Agent 的追问质量**：“红色按钮旁边有没有符号？是实心还是空心？”
- **手册规则的复杂度**：多重条件判断、例外情况、需要交叉引用的条目
- **时间压力**：全球排行榜比的是最快通关时间

### 1.3 设计原则

- **游戏本体是一个网页**，不需要安装任何东西
- **Agent 端不限定工具**，任何能对话的 AI 都能当“专家”
- **手册是一个公开链接**，agent 通过 URL 读取
- **手册格式对人类不友好**，防止人类扮演 agent 角色作弊
- **用户可以优化自己的 agent**，通过 skills/prompt 迭代策略

## 2. 技术架构

### 2.1 总体架构

```markdown
┌─────────────────┐     语音      ┌──────────────────────┐
│   游戏网页        │ ←──────────→ │  任意 AI 工具          │
│  (玩家操作)       │              │  (ChatGPT/Claude/     │
│ bombsquad.amio.fans│             │   Gemini/OpenClaw)    │
└────────┬────────┘              └──────────┬───────────┘
         │                                  │
         │ 每日谜题数据                       │ 读取手册
         │                                  │
┌────────┴────────┐              ┌──────────┴───────────┐
│   游戏服务器      │              │   手册网页             │
│  (谜题生成/排行榜) │              │  bombsquad.amio.fans/ │
│                  │              │  manual/2026-03-11    │
└─────────────────┘              └──────────────────────┘
```

### 2.2 游戏网页（玩家端）

一个纯前端网页应用。玩家打开即玩，不需要账号（排行榜可选注册）。

**显示内容**：
- 2D 谜题场景（多个模块组成的“炸弹面板”）
- 计时器（从开始到完成的用时）
- 操作交互区（点击、拖拽、输入数字等）
- 完成后显示用时和全球排名

**不包含**：语音功能、AI 接口、手册内容。这些全部在 agent 端。

### 2.3 手册网页（Agent 端）

每日手册是一个独立的 URL，如 `bombsquad.amio.fans/manual/2026-03-11`

用户把这个链接发给自己的 AI 工具，AI 工具访问链接读取内容。

#### 手册格式：对 AI 友好，对人类不友好

> [!important] 关键设计决策
> 手册不用 JSON。JSON 虽然 AI 能读，但结构太清晰，人类一样能读。需要一种 AI 能高效处理但人类很难快速消化的格式。

**推荐方案：YAML + 编码条件树**

```yaml
modules:
  - id: "WR-3X"
    type: "wire_routing"
    rules:
      - condition:
          wire_count: {gte: 4}
          has_color: ["red", "blue"]
          serial_last_digit: {odd: true}
          indicator: {lit: "FRK", present: true}
        action: "cut_wire"
        target: 
          position: "last"
          color: "not:white"
        exception:
          if:
            adjacent_color: ["red", "red"]
          then:
            target:
              position: "first"
              color: "blue"
      - condition:
          wire_count: {gte: 4}
          has_color: ["red", "blue"]
          serial_last_digit: {odd: false}
        action: "cut_wire"
        target:
          position: 2
          color: "any"
```

**为什么这个格式好**：

人类看到这个，需要在多层嵌套中跟踪条件分支、交叉引用例外情况，极其痛苦。而 AI 天然擅长解析结构化数据、匹配多重条件、处理嵌套逻辑。

同时，手册中可以加入大量“干扰数据”——不相关的规则条目、红鲱鱼条件、需要跨模块引用的变量。这让人类即使能读也无法快速定位，但 AI 的检索和过滤能力不受影响。

#### 手册网页的人类访问体验

人类用浏览器打开手册 URL 时，首先看到一个友好的提示页面：“这份手册是为 AI 设计的，人类无法有效阅读。请将此链接发送给你的 AI 助手，让它来当你的拆弹专家。” 提示下方才是渲染为 anti-human 样式的手册数据（极小字号、无对比度、无分段）。AI 工具通过 API 或 web fetch 读取时，获得的是清晰的结构化数据。

### 2.4 通信方式

**游戏网页和 Agent 之间没有技术连接。** 沟通完全通过用户的物理语音进行：

1. 玩家对着手机/电脑说话描述场景
2. AI 工具（ChatGPT/Claude/Gemini 的语音模式）听到后回答
3. 玩家听到 AI 的语音指导后在游戏网页上操作

这意味着：
- **零集成成本**——不需要和任何 AI 平台做 API 对接
- **天然兼容所有 AI 工具**——只要支持语音对话就能玩
- **OpenClaw 等 agent 工具**也能用——通过 WhatsApp/Telegram 语音消息

## 3. Roguelike 每日挑战系统

### 3.1 核心机制：手册不变，谜题每次重生

每天发布一份新手册。手册在当天 24 小时内保持不变。

但每次玩家点击“开始”，系统基于当日手册的规则**随机生成一个新的谜题实例**。模块类型和数量固定（确保所需步骤一致），但具体参数（线的颜色、符号的组合、按钮的状态等）每次不同。

**不限挑战次数。** 排行榜记录每个玩家当天的**最快一次**通关时间。

### 3.2 为什么这个设计是对的

**背答案无效。** 你不能记住“剪第二根红线”，因为下一局可能是五根线、颜色完全不同。你只能掌握规则本身——“四根线以上先看序列号奇偶”。这种知识是合法的、值得积累的。

**手册不变 = agent 越来越强。** 同一份手册下，agent 在第一局可能还在熟悉结构，到第三局已经知道关键规则在哪里。尤其在支持长对话的 AI 工具里，agent 会记住之前的沟通模式。人和 agent 都在变强。

**“再来一局”的冲动有出口。** 第一次 8 分钟，复盘优化后第二次 5 分钟，调完 skills 第三次 3 分钟。这个进步曲线本身就有成就感，而且可以立刻验证。

**排行榜有持续竞争。** 不是一锤子买卖。你看到有人 2 分钟通关，你知道是可能的，想磨到更快。一整天都有动力。

### 3.3 谜题生成规则

每个模块类型有一个**参数空间**和一个**复杂度约束**：

- 线路模块：线的数量固定（比如今天是 5 根），颜色和样式随机
- 密码盘模块：轮盘数量固定（比如今天是 3 个），符号随机但保证手册中有唯一解
- 按钮模块：参数随机但保证至少需要 2 次条件判断
- 键盘模块：符号随机但保证手册中有唯一匹配序列

**关键约束：每次生成的谜题难度一致。** 不会出现某次运气好碰到简单配置的情况。

### 3.4 练习关卡

一个永久可用的固定谜题 + 固定手册，不限次数，不计入排行榜。目的：
- 让新玩家熟悉游戏流程
- 测试 AI 工具的连接和语音交互
- 调试 skills 文件

练习关卡的手册比每日挑战简单，规则条目更少、条件分支更浅。

### 3.5 全球排行榜

- 记录每个玩家当天的**最快一次**通关时间
- 每日重置
- 显示：用时、使用的 AI 工具（可选标注）、尝试次数、排名
- 排行榜成绩需要完整操作序列验证（操作顺序和时间戳必须合理）

## 4. 兼容性设计

### 4.1 支持的 AI 工具

| 工具 | 接入方式 | 语音方式 |
|------|---------|---------|
| **Claude** | 把手册 URL 发到对话中 | Claude 内置语音模式 |
| **ChatGPT** | 把手册 URL 发到对话中 | ChatGPT Voice |
| **Gemini** | 把手册 URL 发到对话中 | Gemini Live |
| **OpenClaw** | 配置手册 URL 为 skill 资源 | WhatsApp/Telegram 语音 |
| **其他 Agent** | 任何能读 URL + 语音对话的工具 | 各自的语音方案 |

### 4.2 标准化 Prompt

游戏网页提供一个“开始游戏”页面，自动生成当天的 prompt 模板：

```markdown
你是一个拆弹专家。你的搭档正在面对一个炸弹，需要你通过阅读操作手册来指导她拆弹。

操作手册在这里：[URL]

请先阅读手册，然后告诉你的搭档你准备好了。她会用语音描述她看到的东西，你需要在手册中找到对应的规则，告诉她该怎么操作。

注意：
1. 她可能描述不精确，你需要主动追问关键细节
2. 有些规则有例外情况，请仔细核查所有条件
3. 尽量简洁明了地给出指令，时间很宝贵
4. 如果你不确定，告诉她你需要她再确认什么信息
5. 手册中有很多干扰规则，只关注她描述的模块类型
6. 她可能会连续挑战多次，每次谜题不同但手册规则相同
```

用户复制这段 prompt + URL，粘贴到自己的 AI 工具里，就可以开始玩。

### 4.3 手册的 URL 规范

手册 URL 提供多种格式以兼容不同 AI 工具的读取方式：

- `bombsquad.amio.fans/manual/2026-03-11` — 默认，返回 YAML
- `bombsquad.amio.fans/manual/2026-03-11?format=yaml` — 显式 YAML
- `bombsquad.amio.fans/manual/2026-03-11?format=text` — 纯文本描述版
- `bombsquad.amio.fans/manual/2026-03-11?format=html` — 人类不友好的网页渲染

## 5. Agent 训练 / Skills 系统

### 5.1 核心理念

不同的 agent 读同一份手册，表现差异巨大。差异来自：
- 能不能快速定位到正确的规则条目
- 会不会主动追问关键信息
- 指令是否简洁明确
- 遇到模糊情况的判断策略

用户可以通过编写 **skills 文件**来优化自己 agent 的表现。这个 skills 文件本质上就是一份补充 prompt，教 agent “怎么当一个好的拆弹专家”。

### 5.2 Skills 文件结构

一个 skills 文件是一段文本，用户可以和主 prompt 一起发送给 agent。示例：

```markdown
# BombSquad 拆弹专家策略 v3.2

## 沟通策略
- 我的搭档描述场景时经常忽略位置信息，总是第一时间追问"从左到右的顺序是什么"
- 她区分不清"条纹"和"渐变"，如果她说"好像有花纹"，追问"是一条一条的线还是颜色慢慢变化的"
- 给指令时用"从左数第X个"而不是"倒数第X个"，她处理"倒数"会慢一拍

## 手册检索策略
- 先确认模块类型，再进入对应章节
- 检查完主规则后，一定要检查 exceptions 字段
- 遇到跨模块条件时，先问清其他模块的状态再给答案

## 我的习惯
- 我操作按钮模块最快，线路最慢
- 我经常忘记告诉你指示灯的状态，你要主动问
- 如果我说"等一下"，不要继续说话，我在操作
```

### 5.3 赛后复盘：与 Agent 一起迭代

每次游戏结束后，游戏网页生成一份**结果摘要**（可复制的文本），包含：
- 总用时和每个模块的用时拆解
- 操作错误记录（哪一步做错了、导致了什么后果）
- 关键沟通节点时间线（何时描述、何时收到指令、何时执行）

用户将这份摘要发给 agent，和 agent 一起讨论：“刚才密码盘那里为什么卡了这么久？”“你觉得我的描述哪里不够清楚？”“下次遇到类似情况你应该怎么引导我？”

Agent 根据这次的复盘对话，和用户一起优化 skills 文件。这个过程本身就是一次有意义的人机协作——不是人单方面调参数，是双方一起反思怎么配合得更好。

在 Roguelike 模式下，这个循环变得非常紧凑：打一局 → 复盘 → 优化 skills → 立刻再打一局验证 → 再复盘。每次循环都能看到可量化的进步。

### 5.4 Skills 分享社区

用户可以在 AMIO 社区（或 ClawHub 等平台）分享自己的 skills 文件。形成一个“拆弹专家策略”的 UGC 生态：

- 按 AI 工具分类（Claude 版策略、ChatGPT 版策略……）
- 按排行榜战绩排名（用这个 skills 的最好成绩是多少）
- 社区评分和评论

## 6. 反作弊设计

### 6.1 防止人类代替 Agent 读手册

- 手册格式为嵌套 YAML/结构化数据，人类极难快速阅读
- 手册网页渲染为 anti-human 样式（极小字号、无对比度、无分段），上方有友好提示
- 每日手册包含 200+ 条规则，其中只有 15-20 条与当日谜题相关，人类无法快速筛选

### 6.2 Roguelike 排行榜公平性

- 每次挑战生成的谜题难度一致（相同模块数、相同步骤数、相同条件复杂度）
- 排行榜记录最快一次，但同时显示尝试次数（社区可以看到谁是“一次过”的高手 vs 磨了 20 次的优化者）
- 排行榜成绩需要完整操作序列验证（操作顺序和时间戳必须合理）

### 6.3 防止脚本自动化

- 谜题场景包含需要视觉识别的元素（符号、颜色、空间位置）
- 某些操作需要拖拽/长按等非简单点击
- 场景中的视觉元素随机化渲染（同一个符号的位置、大小、旋转角度每次不同）

## 7. MVP 开发路径

> [!note]
> 详细的 MVP 设计和开发计划见 [[AmiClaw_MVP]]

### Phase 1：MVP 核心验证（3 周）

验证“人说话 + AI 读手册 + 人操作”这个循环好不好玩。包含练习关卡 + 一个每日挑战（Roguelike 无限次）+ 排行榜。

### Phase 2：内容扩展（4 周）

- 增加到 6 种模块类型
- 自动谜题 + 手册生成系统
- Skills 分享社区
- 操作回放功能

### Phase 3：生态化（4 周）

- 与 AMIO 鲨之星集成（排行成绩 → 星球贡献）
- 跨游戏 Skills 生态
- 更多游戏类型（探索未知世界、案件推理等）

## 8. 与 AMIO 的关系

BombSquad 是 AMIO 生态中的第一个“人机协作游戏”。它验证的核心假设是：

> **人和 AI 通过语音协作解决问题，这件事本身好不好玩？**

如果验证通过，这个协作模式可以扩展到更多游戏类型（探索未知世界、案件推理、塞尔达式谜题地牢）。所有这些游戏都共享同一套 agent skills 生态和星球贡献系统。

BombSquad 不依赖 3D 世界、不依赖 agent 培养系统、不依赖多人匹配。它是一个独立的、轻量的、能快速上线的产品。同时它天然兼容所有主流 AI 工具，不需要用户迁移到 AMIO 自己的 AI 平台。

**它验证的不是“AMIO 的 AI 好不好用”，而是“人和 AI 一起玩游戏好不好玩”。** 这是更根本的问题。
