# AmiClaw BombSquad MVP：开发与设计文档

> **平台定位**：BombSquad (`bombsquad.amio.fans`) 是 AmiClaw 平台 (`claw.amio.fans`) 上的第一款游戏，用于验证”人机协作玩游戏”这个核心假设。

> **MVP 目标**：用一个网页 + 一份手册 + Roguelike 无限挑战，验证”人和 AI 通过语音协作拆弹”这件事好不好玩。

## 1. MVP 范围

### 1.1 做什么

- 一个游戏网页（纯前端，托管在 Cloudflare Pages）
- 一个手册网页（静态页面，同样托管）
- 一个练习关卡（固定谜题 + 固定手册，不限次数，不计排行）
- 一个每日挑战（每天一份手册，每次随机生成新谜题，不限次数，记录最快成绩）
- 谜题随机生成器（前端生成，确保每次步骤数一致）
- 一个全球排行榜（Cloudflare Workers + KV）
- 赛后结果摘要（可复制文本，用于和 agent 一起复盘）
- 一份标准 prompt 模板 + 一份示例 skills 文件

### 1.2 不做什么

- 不做语音系统（用户自带 AI 工具的语音）
- 不做 AI 接口对接（零集成）
- 不做账号系统（排行榜用昵称 + 设备指纹）
- 不做难度分级（只有一个难度）
- 不做 Skills 社区（用户自己管理 skills 文件）
- 不做自动手册生成（MVP 阶段手动设计手册）

### 2.2 谜题场景设计（MVP 版）

MVP 包含 4 个模块类型。每次挑战使用全部 4 个，顺序固定。

#### 模块 A：线路（Wire Routing）

**视觉**：4-5 根彩色线，从左侧端子连接到右侧端子。线可能有交叉。每根线有颜色（红/蓝/黄/绿/白/黑）和可选的条纹样式。

**交互**：点击一根线 = 剪断。剪错线 → 错误提示 + 该模块重置（重新随机生成，计时不停）。

**随机化参数**：线的数量（4或5）、颜色分配、条纹样式、排列顺序。

**固定约束**：所需步骤 = 1 步（选对一根线剪断）。

**玩家需要描述**：有几根线、每根线的颜色、从上到下的顺序、有没有条纹。

#### 模块 B：密码盘（Symbol Dial）

**视觉**：3 个旋转轮盘，每个轮盘上有 6 个抽象符号。

**交互**：点击轮盘左右箭头旋转，全部停在正确位置后点击“确认”。选错 → 该模块重置。

**随机化参数**：每个轮盘的符号组合（从符号池中随机选取）。

**固定约束**：3 个轮盘，每个 6 个符号，所需步骤 = 3 步（确定 3 个正确位置）。

**玩家需要描述**：每个轮盘上的符号形状。

> [!tip] 最考验沟通的模块
> 抽象符号（如“一个弯曲的箭头上面有个点”）很难用语言精确描述。这恰好是游戏最好玩的部分——人和 AI 之间因为“描述不清”产生的来回确认。Skills 文件中的“符号命名约定”能极大提升效率。

#### 模块 C：按钮（The Button）

**视觉**：一个大按钮，有颜色和文字标签。旁边有一个指示灯和一个数字显示屏。

**交互**：短按或长按。长按时指示灯颜色会变化，需要在特定颜色时松手。

**随机化参数**：按钮颜色、文字、指示灯初始颜色、数字。

**固定约束**：所需步骤 = 1 步（执行正确的按/放操作），但条件判断链路长。

**玩家需要描述**：按钮颜色、文字、指示灯颜色、数字、序列号和电池数。

#### 模块 D：键盘（Keypad）

**视觉**：一个 2x2 的符号网格，每个格子里有一个抽象符号。

**交互**：按正确顺序点击 4 个符号。点错 → 该模块重置。

**随机化参数**：4 个符号从符号池中随机选取。

**固定约束**：所需步骤 = 4 步（按正确顺序点击）。

**玩家需要描述**：4 个符号的形状。

### 2.3 视觉风格

- **整体色调**：深色背景（#1a1a2e），霓虹感的 UI 元素
- **谜题区域**：中央主体，占屏幕 60%
- **状态信息**：顶部（计时器、模式标识）
- **进度条**：底部（模块完成进度）
- **字体**：等宽字体（模拟控制台感）+ 无衬线字体（UI 文字）
- **响应式**：适配手机和桌面。手机竖屏时模块纵向排列
- **成功动画**：模块完成时有简短的“解除”动画（线断开、锁打开等）
- **错误反馈**：屏幕短暂闪红 + 震动（手机端）

## 3. 手册设计

### 3.1 练习手册

固定不变，规则简单，约 30 条规则 + 20 条干扰。用于新手熟悉流程。

### 3.2 每日手册结构

每份手册是一个 YAML 文件，包含：

```yaml
manual:
  version: "2026-03-11"
  
  # 全局变量说明——这些信息玩家能在场景信息栏看到
  global_context:
    serial_number: "场景信息栏中的6位字母数字序列号"
    battery_count: "场景信息栏中的电池数量"
    indicators: "场景信息栏中的指示灯标签和亮灭状态"

  # 有效模块规则（4个，对应当日谜题的4个模块）
  modules:
    wire_routing:
      rules:
        - condition: {wire_count: 4, color_at_last: "red"}
          action: "cut_wire"
          target: {position: "last"}
        - condition: {wire_count: 4, count_blue: {gt: 1}, serial_last_digit: {odd: true}}
          action: "cut_wire"
          target: {position: "first", color: "blue"}
        # ... 15-20 条规则覆盖所有可能的线路配置 ...

    symbol_dial:
      columns:
        - ["Ω", "Ψ", "☆", "Δ", "Ξ", "◊"]
        - ["Ψ", "◊", "Ω", "☆", "Ξ", "Δ"]
        - ["☆", "Ξ", "Δ", "Ψ", "◊", "Ω"]
        - ["Δ", "☆", "◊", "Ξ", "Ω", "Ψ"]
        - ["Ξ", "Δ", "Ψ", "Ω", "☆", "◊"]
        - ["◊", "Ω", "Ξ", "☆", "Ψ", "Δ"]
      rule: "找到包含场景中全部3个符号的列，按该列从上到下的顺序旋转轮盘"

    button:
      rules:
        - condition: {color: "blue", text: "中止"}
          action: {type: "hold", release_on_light: "white"}
        - condition: {battery_count: {gt: 2}, text: "爆炸"}
          action: {type: "tap"}
        - condition: {color: "red", indicator: {label: "FRK", lit: true}}
          action: {type: "tap"}
        # ... 10-15 条规则 ...

    keypad:
      sequences:
        - ["Ω", "Δ", "Ψ", "Ξ", "◊", "☆"]
        - ["☆", "Ξ", "Ω", "Δ", "◊", "Ψ"]
        - ["Ψ", "◊", "☆", "Ω", "Δ", "Ξ"]
        - ["Δ", "Ω", "Ξ", "◊", "Ψ", "☆"]
        - ["◊", "Ψ", "Δ", "☆", "Ξ", "Ω"]
        - ["Ξ", "☆", "◊", "Ψ", "Ω", "Δ"]
      rule: "找到包含场景中全部4个符号的序列，按该序列的顺序点击"

  # 干扰数据：6-8 个假模块的完整规则
  decoy_modules:
    morse_code:
      # ... 完整但无关的摩尔斯电码规则 ...
    maze:
      # ... 完整但无关的迷宫规则 ...
    memory:
      # ... 完整但无关的记忆模块规则 ...
    simon_says:
      # ... 完整但无关的 Simon Says 规则 ...
    # 总计 200+ 条规则，真正需要的只有 40-50 条
```

### 3.3 手册网页实现

**API/fetch 访问**（AI 工具）：返回纯 YAML 文本，结构清晰。

**浏览器访问**（人类）：
- 顶部：友好提示区域（正常字号、正常配色），说明手册是给 AI 读的，附带返回游戏首页的链接
- 底部：YAML 数据渲染为 anti-human 样式（字号 4px、颜色 #888 on #999、无换行无缩进）

## 4. 谜题随机生成器

### 4.1 生成逻辑

前端在用户点击“开始”时，基于当日手册的规则约束随机生成谜题参数：

```javascript
function generatePuzzle(manual) {
  const seed = Date.now(); // 每次不同
  const rng = seedRandom(seed);
  
  return {
    serial_number: generateSerial(rng),      // 随机6位
    battery_count: rng.intBetween(1, 4),     // 1-4个电池
    indicators: generateIndicators(rng),      // 随机指示灯
    modules: [
      generateWireModule(rng, manual.modules.wire_routing),
      generateDialModule(rng, manual.modules.symbol_dial),
      generateButtonModule(rng, manual.modules.button),
      generateKeypadModule(rng, manual.modules.keypad),
    ]
  };
}
```

### 4.2 答案验证

每个模块的正确答案在生成时就确定了——基于生成的参数 + 手册的规则，通过规则引擎计算出唯一正确操作。

```javascript
function solveWireModule(config, manual) {
  // 遍历手册中的 wire_routing 规则
  // 找到第一条匹配当前配置的规则
  // 返回正确答案（剪哪根线）
  for (const rule of manual.modules.wire_routing.rules) {
    if (matchCondition(rule.condition, config)) {
      return rule.target;
    }
  }
}
```

> [!warning] MVP 安全注意
> 答案计算在前端完成，意味着玩家可以通过开发工具看到答案。MVP 阶段接受这个风险——核心目的是验证玩法体验。正式版需要将答案验证移到后端。

### 4.3 难度一致性保证

每次生成的谜题，通过规则引擎验证以下条件：
- 线路模块：答案唯一（不能出现多根线都符合条件的情况）
- 密码盘模块：手册列中有且只有一列包含场景中的全部符号
- 按钮模块：手册规则中有且只有一条匹配当前配置
- 键盘模块：手册序列中有且只有一条包含场景中的全部符号

如果生成的配置不满足上述条件，丢弃并重新生成（概率较低）。

## 5. Prompt 与 Skills 设计

### 5.1 标准 Prompt

```markdown
你是一个拆弹专家。你的搭档正在面对一个炸弹，需要你通过阅读操作手册来指导她拆弹。

操作手册在这里：{MANUAL_URL}

请先阅读完整手册，然后告诉你的搭档你准备好了。

游戏规则：
- 她会用语音描述她看到的炸弹场景
- 你需要在手册中找到对应的规则，用语音告诉她该怎么操作
- 时间越短越好，全球排行榜在等你们
- 她可能会连续挑战多次，每次谜题不同但手册规则相同

重要提示：
1. 她可能描述不精确，你需要主动追问关键细节
2. 有些规则有例外情况，请仔细核查所有条件
3. 尽量简洁明了地给出指令，避免长篇解释
4. 如果你不确定，让她再确认而不是猜测
5. 手册中有很多干扰规则（如摩尔斯电码、迷宫等），只关注她描述的模块类型
6. 别忘了问场景信息栏中的序列号、电池数和指示灯状态——很多规则依赖这些
```

### 5.2 示例 Skills 文件

```markdown
# BombSquad 拆弹策略 v1.0

## 通用原则
- 一次只处理一个模块，处理完再进入下一个
- 给指令前先确认所有条件都已收集齐
- 用"从左数第X根"描述线路位置

## 符号描述约定
我们约定以下简称：
- Ω = "马蹄"
- Ψ = "叉子"
- Δ = "三角"
- ☆ = "星星"
- Ξ = "三横"
- ◊ = "菱形"
如果我说了你不认识的描述，先复述确认

## 开局流程
每次开始时，我会先报：
1. 序列号
2. 电池数
3. 指示灯
你记下来，后面各模块都可能用到

## 我的弱点
- 我容易紧张说错颜色，如果组合在手册里找不到，让我重新看
- 长按按钮时多给我1秒提前量再说"松手"
```

## 6. 技术实现

### 6.1 技术栈

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 游戏前端 | React + Vite | 单页应用，纯前端 |
| 谜题渲染 | SVG | 2D 谜题场景，便于随机化渲染 |
| 谜题生成 | 前端 JS | 基于手册规则的参数化随机生成 |
| 手册网页 | 静态 HTML + YAML | 极简，无后端 |
| 排行榜 | Cloudflare Workers + KV | Serverless，低成本 |
| 托管 | Cloudflare Pages | 免费，全球 CDN |

### 6.2 数据流

```markdown
1. 用户打开 bombsquad.amio.fans
2. 用户复制 prompt（含手册 URL）发给 AI 工具
3. AI 工具访问手册 URL，获取 YAML 数据
4. AI 语音说"准备好了"
5. 用户点击"开始挑战"
6. 前端基于当日手册规则随机生成谜题参数
7. 前端基于参数 + 规则计算正确答案
8. 计时开始
9. 用户语音描述 ↔ AI 语音回答（物理层面，游戏不参与）
10. 用户在网页上操作（点击/拖拽）
11. 前端验证操作是否正确
    - 正确 → 下一模块
    - 错误 → 该模块重置（重新生成），计时不停
12. 全部模块完成 → 计时停止
13. 如果是今日最佳 → 提交到 /api/leaderboard
14. 显示结果页 + "再来一局"按钮
```

### 6.3 排行榜 API

```markdown
POST /api/leaderboard
{
  "date": "2026-03-11",

GET /api/leaderboard?date=2026-03-11
返回当日排行榜 top 100
含：昵称、最佳用时、尝试次数、使用的 AI 工具
```

## 8. 验证指标

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| **核心体验** | 第一次通关后点击“再来一局” | 数据统计 |
| **Roguelike 循环** | 平均每人每天尝试 3+ 次 | 数据统计 |
| **进步感** | 第 3 次尝试比第 1 次快 30%+ | 数据统计 |
| **完成率** | >70% 的人能在 10 分钟内完成练习关卡 | 数据统计 |
| **AI 兼容性** | 至少 3 种 AI 工具能正常玩 | 逐一测试 |
| **社交传播** | 有人主动分享成绩或 skills | 观察 |
| **复盘有效** | 复盘后下一次成绩明显提升 | 数据统计 |
